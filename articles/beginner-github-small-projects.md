# 初心者がGitHubに小さな作品を増やして分かったこと
完璧じゃなくても、作ったものを残すだけで成長が見える

## はじめに

最近、GitHubのリポジトリが少しずつ増えてきました。

最初から大きなサービスを作っていたわけではありません。
むしろ、ひとつひとつは小さな作品ばかりです。

- JavaScriptで作ったミニゲーム
- Pythonで試した小さなゲーム
- ピアノアプリ
- ブロック崩し
- インベーダー風ゲーム
- Pac-Man風の実験
- ジャンプアクション
- AIと一緒に作った開発環境の試作
- ローカルLLMやAIエージェントを使った実験

どれも完璧なものではありません。

コードもまだ整理しきれていないものがあります。
READMEが薄いものもあります。
今見返すと、直したいところもたくさんあります。

でも、それでもGitHubに残してきてよかったと思っています。

この記事では、初心者としてGitHubに小さな作品を増やしてきた中で感じたこと、学んだこと、そして「完璧じゃなくても公開していい」と思えるようになった理由を書いてみます。

## GitHubは最初、少し怖かった

最初、GitHubは少し怖い場所に感じていました。

自分の中では、GitHubという場所は「すごいエンジニアが使う場所」というイメージがありました。

- 大きなOSSプロジェクト
- 綺麗に整理されたコード
- しっかり書かれたREADME
- IssueやPull Requestが飛び交う世界
- 企業やプロの開発者が使う場所

そんな印象が強くて、自分のような初心者が小さな作品を置いていいのか、少し迷っていました。

「このコード、見られたら恥ずかしいな」
「もっと綺麗にしてから公開した方がいいのかな」
「こんな小さいものをリポジトリにしていいのかな」
「READMEもちゃんと書けてないしな」

そんなことをよく考えていました。

でも、実際に少しずつ使っていくうちに、GitHubの見え方が変わってきました。

GitHubは、完成品だけを置く場所ではなく、
**自分の学習の足跡を残す場所にもなる。**

そう思えるようになりました。

## 小さな作品でも、ちゃんと意味がある

初心者のころは、どうしても「すごいものを作らないといけない」と思いがちです。

- 本格的なWebサービス
- 綺麗なポートフォリオ
- 認証機能つきのアプリ
- データベースを使ったシステム
- ReactやNext.jsを使ったモダンな構成
- 見た目も整った完成品

もちろん、そういうものを作れるようになるのは素晴らしいことです。

でも、最初からそこを目指しすぎると、途中で苦しくなることがあります。

僕の場合は、まず小さなものを作るところから始めました。

- ボタンを押したら音が鳴る
- キャラクターが動く
- 敵が出る
- スコアが増える
- 画面が切り替わる
- キー入力に反応する
- 簡単なUIを表示する

本当に小さな動きです。

でも、自分が書いたコードで画面が動くと、やっぱり嬉しいです。

特にゲームや小さなアプリは、結果がすぐ見えるので楽しいです。

- コードを書いた
- ブラウザで開いた
- キャラクターが動いた
- 音が鳴った
- 当たり判定が動いた

それだけで、「あ、自分でも作れるんだ」という感覚が少し生まれます。

この感覚は、初心者にとってかなり大事だと思っています。

## GitHubに置くと、学習が「見える形」になる

GitHubに作品を置く一番のメリットは、**学習の記録が見える形で残ること**だと思います。

勉強だけしていると、自分がどれくらい進んだのか分かりにくいです。

- 動画を見た
- 記事を読んだ
- コードを写経した
- エラーを直した

もちろん、それも学習です。

でも、時間が経つと「何をやったんだっけ？」となりやすいです。

GitHubにリポジトリとして残しておくと、あとから見返せます。

- このころはJavaScriptのイベントを試していた
- このころはPythonでゲームループを書いていた
- このころはCSSの配置で悩んでいた
- このころはGitHub Pagesで公開しようとしていた
- このころはAIと一緒にコードを直していた

そういう記録が残ります。

ひとつひとつのリポジトリは小さくても、並べてみると「自分はちゃんと進んできたんだな」と感じられます。

これは、想像以上に大きかったです。

## READMEを書くことで、自分の理解も整理される

GitHubに置くようになってから、READMEの大切さも少しずつ分かってきました。

最初はREADMEを書くのが少し面倒に感じていました。

- 何を書けばいいのか分からない
- 説明するほどの作品じゃない気がする
- スクリーンショットも用意していない
- 動かし方を書くのも難しい

でも、READMEを書くと、自分が何を作ったのか整理できます。

例えば、簡単なミニゲームでも、

- これは何をする作品なのか
- どんな技術を使ったのか
- どうやって動かすのか
- どこで詰まったのか
- 何を学んだのか

こういうことを書くだけで、作品が「ただのファイル」から「学習記録」に変わります。

完璧なREADMEでなくてもいいと思います。

最初は短くても大丈夫です。

> 「JavaScriptで作ったシンプルなジャンプゲームです」
> 「HTML/CSS/JSの練習として作りました」
> 「キー入力と当たり判定を学ぶために作りました」
> 「まだ途中ですが、少しずつ改善していきます」

このくらいでも十分です。

大事なのは、未来の自分が見返したときに分かること。

- 「これは何だったのか」
- 「なぜ作ったのか」
- 「何を学んだのか」

それが残っているだけで、かなり助かります。

## 完成度より「残すこと」が大事だった

最初は、完成してから公開しようと思っていました。

でも、実際には「完成」の基準がよく分からなくなります。

- もっとUIを綺麗にしたい
- コードを整理したい
- 機能を増やしたい
- バグを直したい
- READMEを書き直したい
- スクリーンショットを追加したい

そう考えているうちに、公開するタイミングを逃してしまいます。

もちろん、公開するなら最低限動く状態にした方がいいです。
でも、初心者の学習記録としては、完璧を待ちすぎなくてもいいと思うようになりました。

- 途中の作品でも、学びがあります
- 小さな実験でも、意味があります
- 未完成でも、次に直す場所が見えます

むしろ未完成だからこそ、あとから成長が分かります。

昔のコードを見て「ここはもっとこう書けるな」と思えるなら、それは自分が成長した証拠です。

最初から完璧なコードを書ける人はいません。

だから、**今の自分のコードを残しておくことには価値があります。**

## AIと一緒に作る時代になった

僕の場合、最近はAIにもかなり助けてもらいながら作っています。

ChatGPTのようなAIに相談したり、ローカルLLMを試したり、OllamaやHermes Agent、Kiro CLIのようなツールを触ったりしながら、自分用のAI開発環境も作っています。

現在は、ローカルAI司令室のようなものとして「SCORPION BRAIN」も開発中です。

- AIにコードを書いてもらうこともあります
- エラーの原因を一緒に探してもらうこともあります
- READMEの文章を整えてもらうこともあります
- 次に何を直すべきか相談することもあります

でも、AIに全部丸投げするというよりは、
**一緒に考えながら進めている感覚**に近いです。

- 自分が作りたいものを話す
- AIが案を出す
- 実際に動かす
- エラーが出る
- 一緒に直す
- また試す

この繰り返しです。

AIは便利ですが、最終的に「何を作りたいのか」を決めるのは自分です。

そして、GitHubに残すことで、AIと一緒に進めた開発の記録も見えるようになります。

「AIと一緒に作ったから価値がない」のではなく、
「AIを使って、自分が何を形にしたのか」が大事なのだと思っています。

## 小さなリポジトリが増えると、自分の方向性が見えてくる

最初は、とにかく思いついたものを作っていました。

- ゲームを作りたい
- ピアノアプリを作りたい
- AI相棒を作りたい
- サイバーパンクなUIを作りたい
- ターミナルで動くAI環境を作りたい
- ローカルLLMを使ってみたい
- 自分だけの開発室を作りたい

その時々で、面白そうと思ったものを作っていました。

最初はバラバラに見えました。

でも、リポジトリが増えてくると、自分の興味の方向が少し見えてきます。

僕の場合は、

- AI相棒
- ローカルLLM
- 個人開発
- ゲーム制作
- サイバーパンクなUI
- ターミナル環境
- 自分だけのAI研究室

このあたりに自然と集まってきました。

これは、頭の中だけで考えていたら分からなかったかもしれません。

**作って、残して、並べてみる。**
そうすると、自分が何に惹かれているのか見えてきます。

GitHubは、ただのコード置き場ではなく、**自分の興味の地図にもなる**のだと思いました。

## 初心者がGitHubに置くときに意識してよかったこと

ここからは、実際に初心者としてGitHubに作品を置いてみて、意識してよかったことを書いてみます。

### 1. リポジトリ名は分かりやすくする

最初は勢いで名前をつけがちですが、あとから見返すと分かりやすい名前の方が便利です。

例えば、

- `piano-app`
- `breakout-game-js`
- `pacman-python`
- `jump-action-game-js`
- `python-mini-game`

こういう名前だと、何の作品かすぐ分かります。

かっこいい名前も良いですが、学習記録としては分かりやすさも大事だと思いました。

### 2. READMEは短くても書く

READMEは、短くてもあるだけで印象が変わります。

最低限、

1. 何を作ったか
2. 使った技術
3. 動かし方
4. 学んだこと

この4つがあると見返しやすいです。

最初から長いREADMEを書こうとすると大変なので、まずは数行でも大丈夫です。

### 3. スクリーンショットがあると伝わりやすい

コードだけだと、何を作ったのか伝わりにくいことがあります。

ゲームやアプリなら、スクリーンショットが1枚あるだけでかなり分かりやすくなります。

READMEに画像を貼るのは少し面倒に感じるかもしれませんが、作品として見せるなら効果は大きいです。

### 4. 完璧に整理しようとしすぎない

リポジトリを綺麗に整理するのは大事ですが、最初から完璧を目指すと止まりやすいです。

1. まずは置く
2. あとから直す
3. READMEを少し足す
4. ファイル名を整理する
5. 説明を追加する

このくらいの感覚でいいと思います。

### 5. 学んだことを一言でも残す

作品ごとに「学んだこと」を一言残しておくと、あとで見返したときに役立ちます。

例えば、

- JavaScriptのイベント処理を学んだ
- canvasの描画を試した
- キー入力の処理を理解した
- Pythonでゲームループを書いた
- GitHub Pagesで公開する流れを試した

こういうメモがあるだけで、自分の成長ログになります。

## GitHubとZennをつなげると、さらに記録になる

最近は、GitHubだけでなくZennやnoteにも記録を残していこうと思っています。

- GitHubはコードを置く場所
- Zennは技術メモを書く場所
- noteは体験や考えを書く場所

そんなふうに分けると、学習の記録がかなり整理しやすくなります。

例えば、

1. GitHubに小さなゲームを置く
2. Zennで「このゲームを作って学んだこと」を書く
3. noteで「作ってみて感じたこと」を書く

こうすると、ひとつの作品がただのコードではなく、学習記録や発信の材料になります。

初心者のうちは、発信することに抵抗があるかもしれません。

でも、発信は必ずしも「教える」ためだけのものではありません。

- 自分の理解を整理する
- あとから見返せるようにする
- 同じところで詰まっている人の助けになる
- 今の自分の考えを残す

そういう目的でも十分価値があると思います。

## 今後やりたいこと

これからは、GitHubのリポジトリを少しずつ整理していきたいです。

- READMEを増やす
- スクリーンショットを追加する
- 動かし方を書く
- Zenn記事とリンクさせる
- 小さな作品をもう少し見やすく並べる
- AI開発環境やSCORPION BRAINの記録もまとめる

まだまだ途中です。

でも、途中だからこそ残せることがあります。

**完成してから書くのではなく、作りながら書く。**
**理解してから書くのではなく、理解していく途中も書く。**

その方が、初心者の自分に近い言葉で残せると思っています。

## まとめ

GitHubに小さな作品を増やしてきて分かったことは、
**「完璧じゃなくても、作ったものを残すだけで成長が見える」**
ということです。

- 最初から大きな作品を作らなくてもいい
- 綺麗なコードじゃなくてもいい
- READMEが短くてもいい
- まだ途中でもいい

**小さく作って、残して、あとから見返す。**

それだけでも、ちゃんと前に進んでいることが分かります。

初心者にとって、これはかなり大事なことだと思います。

自分の成長は、毎日見えるわけではありません。
でも、GitHubに作品を残しておくと、あとから振り返ったときに気づけます。

- 「あのころより少し分かるようになった」
- 「あのころより作れるようになった」
- 「あのころより説明できるようになった」

その積み重ねが、自信になります。

僕もまだ学習中です。
まだ分からないことはたくさんあります。
作りたいものに技術が追いつかないこともあります。

でも、それでも作っていきます。

小さな作品を増やしながら、AIと一緒に、自分だけの開発環境や作品を少しずつ育てていきたいです。

GitHubは、その足跡を残す場所として、これからも使っていこうと思います。

## 初心者向けGitHub公開チェックリスト

最後に、初心者が小さな作品をGitHubに置くときのチェックリストをまとめておきます。

- [ ] リポジトリ名は分かりやすいか
- [ ] READMEに「何を作ったか」を書いたか
- [ ] 使用技術を書いたか
- [ ] 動かし方を書いたか
- [ ] スクリーンショットを載せられるか
- [ ] 学んだことを一言でも書いたか
- [ ] 完璧を待ちすぎていないか
- [ ] あとから直せる前提で公開できているか

完璧じゃなくても大丈夫です。

まずは小さく作って、残してみる。
それだけでも、未来の自分にとって大切な記録になると思います。
