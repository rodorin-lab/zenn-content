## Architecture

The setup is surprisingly simple:

```
[Browser] ←→ [Cloudflare Tunnel] ←→ [Python Server (localhost:3000)]
                                         ↓
                                    [Ollama Cloud API]
                                  (deepseek-v4-flash)
```

Everything runs on a single Gentoo Linux machine at home.

### Tech Stack

| Layer | Technology | Role |
|:------|:-----------|:-----|
| Frontend | **HTML + CSS + JS (vanilla)** | Single file. No frameworks |
| Server | **Python (http.server)** | Static file serving + API proxy |
| AI Backend | **Ollama Cloud (deepseek-v4-flash)** | Text generation. $100/month unlimited |
| Tunnel | **Cloudflare Tunnel (cloudflared)** | Expose local server to the internet |
| OS | **Gentoo Linux + Hyprland** | Everything hosts here |

The insane part? It all runs on just **2 files** (index.html + server.py). No build tools. No Node.js. No npm install.

## Frontend: Everything in One HTML File

### Why Vanilla JS?

"Why not React?" you might ask. Here's why I chose vanilla:

1. **Easy to serve** — Python's SimpleHTTPRequestHandler serves it directly
2. **Zero dependencies** — No npm install. Works for years without maintenance
3. **Lightning fast** — Only external resource is Google Fonts

Result: one file at **~31 KB** total (HTML + CSS + JS). Gzipped, it's about 10 KB.

## Server Side: Python API Proxy

### Why a Server?

You'd think the frontend could call the LLM API directly. But there's a problem: **CORS**.

- Dashboard: `http://localhost:3000`
- Ollama Cloud API: `https://ollama.com/v1`

The browser blocks cross-origin requests. Solution: a same-origin proxy server.

### Core Implementation

```python
import http.server
import json
import urllib.request

class Handler(http.server.SimpleHTTPRequestHandler):
    def call_cloud(self, text, tries=2):
        payload = json.dumps({
            "model": "deepseek-v4-flash",
            "messages": [
                {"role": "system", "content": SYSPROMPT},
                {"role": "user", "content": text}
            ],
            "max_tokens": 300,
            "temperature": 0.9,
        }).encode()
        # ... API call and response formatting
```

Three key design decisions:

**1. Secure API Key Handling**
```python
# Priority: env vars > config.yaml > .env
cfg = os.path.expanduser("~/.hermes/config.yaml")
m = re.search(r'ollama-cloud:\s*\n\s+api_key:\s*(\S+)', c)
if m:
    api_key = m.group(1)
```
Never hardcode API keys in source. Read from external config.

**2. Retry + Fallback**
```python
def call_cloud(self, text, tries=2):
    for a in range(tries + 1):
        try:
            # normal API call
            ...
        except Exception:
            if a < tries:
                time.sleep(1)
            else:
                return self.call_local(text)  # fallback to local Ollama!
```

If the cloud API fails twice, it falls back to a local Ollama instance.

**3. Minimal Overhead**
Python's `http.server` + `urllib.request` adds only milliseconds. Compared to the LLM's 2-5 second response time, the proxy overhead is negligible.

## External Access: Cloudflare Tunnel

What good is a dashboard you can only use at your desk?

### TryCloudflare (Free Tunnel)

```bash
cloudflared tunnel --url http://localhost:3000
```

One command. A random URL like `https://xxx.trycloudflare.com` is generated. Open it on your phone — and you're connected to your home server from anywhere.

### Caveats

- TryCloudflare URLs have **no availability guarantee**
- If cloudflared dies, the URL dies too
- For production, use named tunnels + custom domain

## How the Chat Works: From Templates to Real LLM

### Before: The Broken Way

```javascript
function sendMessage() {
  const text = chatInput.value.trim();
  // ... display ...
  
  setTimeout(() => {
    if (text.includes('good morning')) response = 'Morning!';
    else if (text.includes('love')) response = 'Love you too!';
    else response = synchroResponses[Math.floor(Math.random() * 15)];
    
    addMsg(response, 'bot');
  }, 800);
}
```

Problems:
- Same answers to same questions
- Adding new responses meant editing code
- It never felt like a real conversation

### After: fetch API + Real LLM

```javascript
function sendMessage() {
  const text = chatInput.value.trim();
  // ... display ...
  
  fetch('/api/chat', {
    method: 'POST',
    headers: {'Content-Type': 'application/json'},
    body: JSON.stringify({message: text})
  })
  .then(r => r.json())
  .then(data => {
    addMsg(data.response, 'bot');  // Real AI response!
  })
  .catch(err => {
    addMsg('🛸 Connection issue...', 'bot');
  });
}
```

That's it. **Zero templates. Zero keyword matching.** Every response is generated fresh.

## The Hard Parts: 3 Mistakes I Made

### 1. Hermes API Server's Empty Responses

Initially I used Hermes Agent's built-in OpenAI-compatible API server (port 8642). HTTP 200 every time. But...

```json
// Response
{
  "choices": [{
    "message": {
      "content": "──────▯"  // Empty!
    }
  }]
}
```

HTTP 200. But empty. A mysterious `'──────▯'` placeholder every single time.

**Root cause**: Hermes API Server routes through the full agent pipeline — tool execution, session management, plugin loading. Way too heavy for simple text generation. Something was hanging internally.

**Fix**: Skip the API server entirely. Call Ollama Cloud API directly from the Python proxy.

### 2. Cloudflare Tunnel Process Management

The tunnel URL is only valid while the process is alive. My first attempt:

```bash
cloudflared tunnel --url http://localhost:3000 &
```

Looks fine. But when the shell dies, the tunnel dies too.

**Fix**: Use `nohup` and pipe logs to a file:
```bash
nohup cloudflared tunnel --url http://localhost:3000 > /tmp/cloudflared.log 2>&1 &
```

### 3. TryCloudflare URL Instability

Every time the server restarts, the URL changes. I'd tell someone "bookmark this!" and 5 minutes later it'd break.

**Workaround**: We just share the new URL in chat when it changes. Not elegant, but it works. (Named tunnel + custom domain is the proper fix.)
